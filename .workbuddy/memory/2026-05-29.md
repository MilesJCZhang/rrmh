## 修复：已有推荐码的会员点击"会员建档"仍提示"需要推荐人"

### 问题
用户"张进铖Miles"已被城市合伙人推荐，服务端 `users.recommendedBy` 有值，
但点击"会员建档"仍提示"需要推荐人才能建档"。

### 根因
`auth.service.js` 的 `syncUserData()` 在登录/profile 同步时，把 `recommendedBy` 写入了 `user_info.referrerId`，
但 **从未调用 `setReferrerId()` 去写入 `referrer_id` Storage key**。

而 `getReferrerId()` 读取的是独立的 `referrer_id` Storage key（非 `user_info.referrerId`），
所以对已登录用户而言，`getReferrerId()` 永远返回 null。

### 修复
在 `syncUserData()` 中，当 `hasReferrerId` 为 true 时，新增调用：
- `setReferrerId(referrerIdValue)` — 写入 `referrer_id` Storage + globalData
- 如果有 `referrerName`，同步写入 `referrer_info`

### 影响范围
- `services/auth.service.js`：`syncUserData()` 新增推荐人同步逻辑
- 版本号：v1.0.8 → v1.0.9

### 补充修复（v1.0.10）
上一步修复只解决了**新登录用户**的问题。老用户（已登录未重新触发 syncUserData）的 `referrer_id` Storage key 仍然为空。
进一步在 `getReferrerId()` 中增加兜底逻辑：当 `referrer_id` 和 `globalData.referrerId` 都为空时，
回退到 `userInfo.referrerId` 读取，找到后自动回填到 Storage。
这样老用户无需重新登录即可修复。

### 根因修正（v1.0.10 服务端）
前端两层修复后仍然无效，深入排查发现**真正的根因在服务端**：
`wechat-login` 接口返回的 `user` 对象中缺少 `recommendedBy` 字段。
该接口返回的字段只有 `id/nickname/avatar/phone/role/roleList/recommendCode/hasProfile/hasAvatar/isVerified`，
导致 `app.js` 的 `login()` → `_syncUserData(data.user)` 从未获得 `recommendedBy` 值，
进而 `syncUserData` 中的 `hasReferrerId` 始终为 false，`userInfo.referrerId` 从未被写入。

**修复**：在 `src/routes/miniAuth.ts` 的 wechat-login 成功响应中新增 `recommendedBy: user.recommendedBy`。
**影响**：所有用户下次登录时会拿到 `recommendedBy`，前端 `syncUserData` 会自动写入 `referrer_id` Storage + `userInfo.referrerId`。
**已部署**：`npm run build && pm2 restart renrenmeihao-api`
