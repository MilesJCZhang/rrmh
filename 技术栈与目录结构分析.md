# 人人媒好相亲小程序 — 技术栈与目录结构全面分析

> 分析时间：2026-07-13
> 当前版本：小程序 `v1.0.24`，管理后台 `v1.0.10`

## 一、整体架构概览

人人媒好是一个同城兴趣社交 / 相亲撮合平台，采用**微信小程序 + 后端 API + 独立管理后台**的三层架构。项目根目录并非单一应用，而是把前端、两套后端、管理后台、数据库脚本、运维脚本和大量文档**混合在同一个仓库**中，这是理解目录结构的前提。

```
用户端（微信小程序 WXML/JS）  ──HTTPS──▶  后端 API 服务（Express / Prisma）
                                            │
管理后台（React + AntD）       ──HTTPS──▶   ├─ 生产后端 S1：TypeScript + Prisma + MySQL（部署在服务器）
                                            └─ 本地后端 S2：JavaScript + Express + better-sqlite3（server.js）
```

注意：仓库里同时存在**两套后端形态**——根目录的 `server.js` + 一大批 `routes_*.js`（JavaScript，SQLite，本地开发态 S2），以及 `src/`（TypeScript，仅含 `routes/user.ts` 与 `services/score.service.ts` 两个片段，对应生产态 S1）。真正运行在生产服务器的是 S1（TypeScript + Prisma + MySQL），本地 `src/` 只有两个参考文件，并未包含完整后端源码。

## 二、技术栈详解

### 2.1 小程序前端（主交付物）

| 维度 | 选型 |
|------|------|
| 开发框架 | 微信原生小程序（WXML + WXSS + JavaScript），未使用 Taro / uni-app 等跨端框架 |
| 基础库版本 | libVersion `3.15.1`（project.config.json），ES6/ES7 增强编译已开启 |
| UI 组件库 | `weui-miniprogram` ^1.2.3 |
| 样式体系 | 自研"国风"设计令牌系统（`app.wxss`，以 CSS 变量定义中国红 × 青瓷蓝配色、阴影、圆角、按钮/卡片/标签组件类） |
| 网络层 | 自研 `utils/request.js`：兼容"云托管 callContainer"与"传统 wx.request"双模式，内置 Bearer Token 注入、401 自动刷新 + 静默重新登录、GET 参数序列化、请求体脏数据清洗 |
| 分包策略 | 1 个主包 + 6 个分包（activity / matchmaker / partner / social / user / premium），`lazyCodeLoading: requiredComponents` 已开启 |
| 代码分包大小 | 主包 9 个页面，分包共约 35 个页面（pages/ + subpackages/ 下约 30 个目录） |

### 2.2 后端 API（两套并存）

本地开发态 S2（`server.js`，`package.json` 描述）：
- Node.js + **Express 5.2.1** + **better-sqlite3 12**（文件型 SQLite 数据库 `renrenmei.db`）
- 依赖：`jsonwebtoken`、`multer`（文件上传）、`qrcode`（普通二维码）、`xml2js`/`js2xmlparser`（微信支付回调 XML）、`pdfkit`（PDF 导出）、`axios`/`node-fetch`、`cors`、`dotenv`、`express-validator`
- 入口 `server.js`，路由拆分为 `routes_user.js` / `routes_salon.js` / `routes_payment.js` / `routes_referral.js` 等约 40 个 `routes_*.js` 文件，认证中间件 `auth-middleware.js`

生产态 S1（部署在服务器 175.24.227.251）：
- TypeScript + **Prisma ORM** + **MySQL**（`renrenmeihao` 库），PM2 管理 `renrenmeihao-api` 进程
- 本地 `src/` 仅有 `routes/user.ts`、`services/score.service.ts` 两个 TS 片段作为参考，完整源码在服务器上

### 2.3 管理后台（admin-panel/）

- **React 18 + TypeScript + Ant Design 5**（`antd` ^5.12），使用 **Create React App（react-scripts 5）** 构建
- 路由：`react-router-dom` ^6.20；图表：`recharts` ^3.8；表格导出：`xlsx` ^0.18.5；样式：`sass`
- 共 22 个页面（`.tsx`）+ 16 个服务层（`.ts`）+ 若干公共组件
- 构建产物部署到服务器 `/var/www/admin/`，通过 Nginx `https://rrmhdate.cn/admin/` 访问

### 2.4 数据库与运维

- `database/migrations/` 下 14 个 `.sql` 迁移脚本（MySQL  schema）+ `referral_system_v2.sql`、`visitor_logs.sql` 等
- 多个 `.py` 运维脚本（`deploy_to_prod.py`、`deploy_workbench.py`、各类 `fix_*.py`）和 `.sh` 部署脚本（deploy_admin_panel.sh 等）
- `jest` + `supertest` 用于后端接口测试（`test_*.js`、`tests/`）

## 四、关键设计模式与工程约定

- **API 端点集中注册**：所有后端路径统一在 `services/api.js` 定义，页面/服务层禁止硬编码，后端改路径只改此文件。
- **配置中心化**：`utils/config.js` 管理环境（dev/prod）、API 基础地址（`prod: https://rrmhdate.cn`）、功能开关 `FEATURES`、版本号单一数据源。上线时改 `ENV='prod'` 并关闭 `DEV_MODE` / `DEV_MOCK_DATA`。
- **统一网络层**：`utils/request.js` 封装 token 注入、401 刷新、静默登录、GET 参数拼接、脏数据清洗，屏蔽微信 native 层 clone 失败的边界坑。
- **角色权限体系**：`constants/roles.js` 定义 7 级角色（admin > city_franchisee > professional_recommender > community_station > partner_matchmaker > public_matchmaker > user），推荐码前缀（LCRG/GYRG/ZYRG/SQZD/CSHH），推荐官角色集合，前后端共用。
- **全局状态**：`app.js` 的 `globalData` 承载 userInfo / token / userRole / 评分 / 推荐人等，并在 `onLaunch/onShow` 做自动登录、隐私授权、scene 推荐人解析。
- **自研设计系统**：`app.wxss` 以 CSS 变量 + 工具类（flex/btn/card/tag/form）构建国风视觉，组件库仅 2 个自定义组件，其余复用 WeUI。

## 五、值得关注的问题与风险

1. **多后端并存、职责不清**：本地 S2（JS+SQLite）与生产 S1（TS+Prisma+MySQL）功能不完全对齐（据历史记录，微信支付、推荐绑定等在 S2 更完整，S1 缺失 `user_referrals` 等表）。`src/` 仅 2 个 TS 片段，本地无法完整构建生产后端，调试与源码管理存在断点。
2. **仓库混杂、技术债集中**：根目录堆积约 40 个 `routes_*.js`、数十个 `fix_*.py`/`*.mjs` 临时脚本、重复备份（`.backup`/`.broken`/`_verify_*.png`）和 20+ 篇 Markdown 文档，不利于维护与新人上手。
3. **前端分包/主包页面注册与实际文件不一致**：部分 `pages/` 下页面（如 `matchmaker/`、`salon/`、`chat/`）既在主包目录又出现在分包，存在重复与未注册页面的冗余。
4. **前端服务层约定与实现有偏差**：`app.js` 中仍直接 `require('./services/api')` 调用，但 `app.json` 的 `lazyCodeLoading` 与组件按需加载已开启，需注意分包按需加载时机。
5. **依赖版本偏旧**：管理后台 `typescript ^4.9`（与 React 18 配合可，但 CRA 5 已步入维护模式）；小程序 `libVersion 3.15.1` 需关注基础库升级兼容性。

## 六、结论

人人媒好是一个**功能覆盖面很广**的社交撮合小程序，技术栈选择务实：前端用微信原生框架 + 自研设计/网络层，后端采用 Node/Express（本地 SQLite 与生产 MySQL 双轨），管理后台用成熟 React + AntD 栈。其架构的最大特点是**模块化分包清晰、配置与 API 高度集中**，但最大隐患是**仓库内多套后端与大量临时脚本长期并存**，建议后续按"前端 / 生产后端 / 管理后台 / 数据库 / 运维脚本 / 文档"做物理目录拆分，并将 S1 完整 TS 源码纳入版本管理。
