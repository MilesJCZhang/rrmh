## 菜单排序逻辑

### 排序原则
1. **用户使用频率排序**：从高频到低频排列
   - 总览仪表盘 > 用户体系 > 业务活动 > 档案资质 > 交易财务 > 渠道合伙人 > 系统配置 > 数据统计
2. **业务操作流排序**：注册 → 活动 → 档案 → 财务 → 渠道 → 配置 → 统计
3. **父子菜单一致性**：同分类下的子菜单按业务操作步骤排序

### 子菜单排序逻辑
- **用户体系**：用户管理 → 实名认证 → 推荐码管理（账号→认证→关系链）
- **业务活动**：活动管理 → 沙龙配置（前台活动→后台配置）
- **档案资质**：档案管理 → 验资托管 → 基金托管（基础档案→资产审核→资金托管）
- **交易财务**：收益明细 → 支付记录 → 被动收益 → 提现管理 → 佣金管理 → 订单管理（收入→支出→资金池）
- **系统配置**：系统设置 → 评分规则（全局配置→业务规则）

## 权限分组思路

### 角色-菜单矩阵（建议）

| 角色 | 可见菜单 |
|------|---------|
| 超级管理员 | 全部 8 个分类 |
| 运营管理员 | 总览仪表盘、用户体系、业务活动、档案资质、渠道合伙人、数据统计 |
| 财务管理员 | 总览仪表盘、交易财务（全部子项）、数据统计 |
| 系统管理员 | 系统配置、数据统计 |
| 客服/审核员 | 用户体系(用户管理+实名认证)、档案资质(档案管理) |

### 权限控制建议
- 在 `Layout.tsx` 中根据 `user.role` 过滤 `menuItems`
- 后端 API 层用 `authMiddleware` + 角色校验双层防护
- 对敏感操作（提现审核、系统设置）增加二次确认

## 适配多后台统一标准化方案

### 多后台现状
- **管理后台**（当前优化）：React + Ant Design Pro，面向内部运营
- **推荐官工作台**（小程序端）：面向推荐官/合伙人

### 菜单标准化建议
1. **统一命名规范**：所有后台使用相同的一级分类名称（总览、用户、业务、财务、配置、统计）
2. **统一路由前缀**：`/v1/admin/*` 统一作为 API 前缀
3. **权限模型统一**：角色-菜单-API 三级联动
4. **菜单配置化**：将 `menuItems` 抽取为 JSON 配置，不同后台加载不同配置
5. **图标一致性**：同一业务模块在不同后台使用相同图标

### 后续迭代建议
1. 将 `Layout.tsx` 的 `menuItems` 抽取为独立配置文件 `config/menus.ts`
2. 支持菜单角色过滤（基于 `user.roleList`）
3. 后端返回菜单配置，前端动态渲染（避免每次部署修改前端）
4. 增加菜单搜索功能（按名称/路径搜索）
