# 公益推荐官 & 联创推荐官 注册业务逻辑深度分析报告

> 分析日期：2026-07-02
> 分析范围：前端小程序 + Express/SQLite 后端（含支付、佣金、角色升级全链路）

---

## 一、总体结论

| 角色 | 流程完整度 | 状态 |
|------|----------|------|
| 公益推荐官 | ~90% | 基本打通，有小瑕疵 |
| 联创推荐官 | ~40% | **存在阻止性 Bug，支付→角色升级链路断裂** |

公益推荐官（免费注册）的核心路径基本完整：用户填写表单 → 后端校验 → 生成推荐码 → 更新角色 → 前端弹窗展示推荐码。联创推荐官（付费注册，¥399）存在一个**阻止性 Bug**：前后端支付类型值不一致，导致支付无法创建或支付后角色不升级。

## 二、公益推荐官（public_matchmaker）全链路分析

### 流程图

```
matchmaker.js: onApplyMatchmaker()
  → wx.navigateTo → apply?type=public
  → apply.js: 填写表单(姓名/性别/年龄/手机号) + 勾选协议
  → onSubmit → _submitPublicApply()
  → applyService.applyPublicMatchmaker(params)
  → POST /v1/apply/public-matchmaker  [routes_apply.js:95]
  → 后端: requireAuth → 参数校验 → 幂等检查 → 生成GYRG推荐码
  → 事务: INSERT referral_codes + UPDATE users.role='public_matchmaker'
  → 返回 { code:0, data:{ role, recommendCode } }
  → 前端: authService.setUserInfo + setUserRole
  → wx.showModal 弹窗展示推荐码 → redirectTo qrcode 页面
```

### 逐环节核查

| 环节 | 文件 | 状态 | 说明 |
|------|------|------|------|
| 入口触发 | matchmaker.js:455 `onApplyMatchmaker()` | OK | 硬编码跳转到 `apply?type=public` |
| 入口触发(二) | matchmaker.js:510 `onApplyPublic()` | OK | 同上 |
| 表单校验 | apply.js:140 `_validatePublic()` | OK | 姓名 ≥2字、性别必选、年龄18-80、手机号校验 |
| 提交调用 | apply.js:184 `_submitPublicApply()` | OK | 调用 applyService |
| API端点 | api.js:70 `APPLY.PUBLIC_MATCHMAKER` | OK | `/v1/apply/public-matchmaker` |
| 后端认证 | routes_apply.js:95 `requireAuth` | OK | JWT Token 验证 |
| 后端校验 | routes_apply.js:102 | OK | real_name ≥ 2字 |
| 幂等检查 | routes_apply.js:107-126 | OK | 已是推荐官则返回已有推荐码 |
| 推荐码生成 | routes_apply.js:51-71 `generateUniqueCode` | OK | GYRG + 4位随机，冲突检测 |
| 事务写入 | routes_apply.js:135-149 | OK | INSERT referral_codes + UPDATE users |
| 状态更新 | apply.js:198-205 | OK | authService.setUserInfo + setUserRole |
| 推荐码展示 | apply.js:210-238 | OK | Modal 弹窗 → redirectTo qrcode |
| 路由挂载 | server.js:232-233 | OK | `/v1/apply` 已挂载 routes_apply |

### 发现的小问题

1. **后端不校验手机号格式** — `routes_apply.js` 未校验 `phone` 字段，前端做了 `/^1[3-9]\d{9}$/` 校验，但若绕过前端直接调 API 可写入任意字符串
2. **申请记录缺失** — 公益推荐官的事务直接写 `referral_codes` 和 `users`，不写 `apply_records` 表（但联创推荐官会写），导致管理后台无申请轨迹
3. **昵称获取按钮样式问题** — `profile.wxss` 中 `.nickname-btn` 的 `padding` 被多次调整（当前为 `0 20rpx`），用户反馈浏览器缓存可能未生效

### 结论：公益推荐官流程基本完整，可以正常使用

## 三、联创推荐官（partner_matchmaker）全链路分析

### 预期流程

```
apply.js: _submitPartnerApply()
  → Step1: POST /v1/apply/partner-matchmaker  提交资料
  → Step2: 弹窗"是否立即支付¥399?"
  → Step3: createPayment({ type: PARTNER_MATCHMAKER_UPGRADE })
     → POST /v1/payment/create  { type: 'partner_upgrade', amount: 399 }
     → 微信支付/模拟支付
  → Step4: 支付回调 POST /v1/payment/notify
     → upgradeUserToPaidTier(orderType, userId)
     → 生成LCRG推荐码 + 角色升级
```

### 🔴 关键Bug #1：前后端支付类型不一致

**位置**：

| 层 | 文件 | 行号 | 使用的值 |
|----|------|------|----------|
| 前端 | `utils/payment.js` | 15 | `'partner_upgrade'` |
| 后端 FEE_MAP | `routes_payment.js` | 118 | `'partner_matchmaker'` |
| 后端 PAID_TIER_CONFIG | `routes_payment.js` | 18 | `'partner_matchmaker'` |
| 后端 commission_engine | `commission_engine.js` | 39 | `'partner_matchmaker'` |
| 存量补录数据 | `server/fix_commission.js` | 18 | `'partner_upgrade'` |

**影响**：

```
场景 A：生产环境（非 DEV_MODE，非 MOCK_PAYMENT）
  → POST /v1/payment/create { type: 'partner_upgrade', amount: 399 }
  → routes_payment.js:182 检查 FEE_MAP['partner_upgrade'] → undefined
  → 返回 400 { code: -1, message: '无效的支付类型' }
  → ❌ 支付无法创建，整个流程中断

场景 B：MOCK_PAYMENT=true（开发/测试环境）
  → 模拟支付成功，订单写入 orders 表 type='partner_upgrade'
  → routes_payment.js:284 调用 upgradeUserToPaidTier(pool, 'partner_upgrade', userId)
  → PAID_TIER_CONFIG['partner_upgrade'] → undefined
  → upgradeUserToPaidTier 直接 return，不做任何操作（静默跳过）
  → ❌ 角色不升级，推荐码不生成

场景 C：微信支付回调（正常支付成功）
  → order.type = 'partner_upgrade'（来自创建订单时的值）
  → routes_payment.js:471 调用 upgradeUserToPaidTier(pool, 'partner_upgrade', userId)
  → 同样静默跳过
  → ❌ 角色不升级，用户付了钱但没有获得联创推荐官身份
```

**这个 Bug 意味着：在任何环境下，联创推荐官的支付→角色升级链路都是断的。**

---

### 🔴 关键Bug #2：申请表单数据丢失

**位置**：`apply.js:254-260` → `routes_apply.js:179-257`

**现象**：前端收集了 4 步表单数据：
- Step 1：姓名、性别、年龄
- Step 2：手机号、微信号
- Step 3：社会活动经验标签（6选 + 补充文本）、推广渠道标签（6选 + 补充文本）
- Step 4：收款方式（微信零钱/银行卡）、收款账号、开户银行

但 `_submitPartnerApply()` 只传递了 `{ real_name, gender, phone, target_role }` 四个字段。后端 `POST /v1/apply/partner-matchmaker` 也只接收（实际未使用）这些字段，不做任何存储——仅写入一条 `apply_records` 记录。

**影响**：用户填写的大量从业资料、推广渠道偏好、收款账户信息全部丢失，审核人员没有依据进行审核。

---

### 🟡 问题 #3：推荐人校验逻辑可绕过

**位置**：`routes_apply.js:202-215`

```javascript
// 检查用户是否有推荐人
const userReferral = db.prepare(
  'SELECT id FROM user_referrals WHERE user_id = ? AND is_locked = 1'
).get(userId);

// 检查是否系统授权
let isSystemAuthorized = false;
const userCode = db.prepare(
  "SELECT referred_by_code FROM referral_codes WHERE referrer_id = ? AND referred_by_code = 'SYSTEM' LIMIT 1"
).get(userId);
if (userCode) isSystemAuthorized = true;
```

这里检查 `referral_codes.referrer_id = userId` — 这意味着要查的是"该用户作为推荐人生成的推荐码"中是否有被系统授权的。但新申请者通常还没有推荐码（因为他正在申请成为推荐官），所以 `isSystemAuthorized` 几乎永远为 `false`。

正确的逻辑应该是检查该用户**被谁推荐**（即该用户绑定了哪个推荐人），而不是检查该用户推荐了谁。这导致没有推荐人绑定的用户直接收到 `code: -1, message: '成为联创推荐官需要先通过推荐码绑定一位推荐人'`。

---

### 🟡 问题 #4：申请人身份兼容性

`_submitPartnerApply()` 中用户可能已经是公益推荐官。根据分润规则，公益推荐官不能直接支付升级为联创推荐官——需要通过 `partner-apply` 页面。但当前 `apply.js` 页面的 `_submitPartnerApply()` 试图直接申请联创推荐官，后端 `partner-matchmaker` 路由的幂等检查只排除了"已是联创推荐官"的情况，没有检查"已是公益推荐官"的升级路径。
