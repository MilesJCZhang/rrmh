# 访客→会员 全业务流程打通分析报告

> 生成时间：2026年7月1日

## 一、执行摘要

经过对前端小程序（pages/、utils/、services/）、后端路由（routes_auth.js、routes_payment.js、routes_member.js、routes_referral.js 等）以及管理后台（admin-panel/）的全面代码审查，**访客到会员的核心业务流程基本打通，但存在一处关键断点**：支付会员建档费（single_registration）后，后端的 `upgradeUserToPaidTier` 函数不会对 `single_registration` 类型执行角色升级操作，导致支付成功后用户的 role 字段始终停留在 `user`，与前端注册页注释中"支付回调已更新 role 为 single"的预期不符。

以下按流程链路逐环节分析。

## 二、数据模型关键字段

| 表 | 关键字段 | 说明 |
|-----|---------|------|
| users | role | 角色标识：`user`（默认）/ `public_matchmaker` / `partner_matchmaker` / `professional_recommender` / `city_franchisee` / `community_station` |
| users | parent_id | 通过微信登录 scene 参数绑定的推荐人 ID（第一次登录即锁定） |
| users | referrer_id | 通过推荐码绑定接口设置的推荐人 ID |
| users | referral_code | 用户自己的推荐码（推荐官自动生成） |
| referral_codes | code / code_type / referrer_id / status | 推荐码表，含类型、关联推荐官、状态 |
| user_referrals | user_id / referrer_id / is_locked | 推荐关系绑定表，唯一约束 user_id，is_locked=1 永久锁定 |
| orders | type / status / user_id | 订单表，type 含 `single_registration`（会员建档费）等 |
| apply_records | target_role / status | 身份申请记录表 |

## 三、完整流程链路分析（7个环节）

### 环节 1：访客进入小程序

**入口路径**：扫码 → `app.js` `_handleScene()` → `parseReferralScene()` → `_autoLogin()`

**处理逻辑**：
- `parseReferralScene()` 支持多种格式：URL 参数（`?referrer_id=30&code=LCRG001`）、scene 编码、纯推荐码字符串、纯数字用户 ID
- 解析成功后将 `referrer_id` 或 `pendingReferralCode` 存入 `globalData` 和 Storage
- `_autoLogin()` 判断是否有 token，有则调用 `GET /v1/user/profile` 获取用户信息，无则触发 `login()`

**访客判断逻辑（存在两套）**：
- `auth.service.isGuest()` = `!isLogin()` — 仅判断是否登录
- `app.globalData.isGuest` = `!hasProfile` — 判断是否登录+建档（`app.js` `_syncUserData()` 设置）

**状态**：✅ 打通。多种入口格式均能正确解析推荐信息并传递到登录流程。

### 环节 2：微信登录与身份初始化

**接口**：`POST /v1/auth/wechat-login`（无鉴权）

**参数**：`{ code, referrer_id, referral_code }`

**后端处理**（routes_auth.js 第79-227行）：

1. 调用微信 `jscode2session` 获取 openid
2. **新用户**：INSERT users 表，同时写入 `parent_id`（来自 scene）和 `user_referrals` 表
3. **已有用户**：若 `parent_id` 为 null 且本次携带 `referrer_id`，补填 `parent_id`（仅允许 null→有值，永久锁定）
4. 调用 `calculateUserRoles()` 计算角色列表
5. 生成 JWT Token（7天有效期）

**返回字段**：`{ token, openid, userId, role, roleList }`

**状态**：✅ 打通。登录链路完整，推荐关系在首次登录时即写入。

**潜在问题**：
- 登录时使用 `parent_id` 字段存储推荐关系，而后端 `routes_payment.js` 佣金计算时从 `users.referrer_id` 读取推荐人——两个字段可能不一致。登录绑定的推荐人在 `parent_id` 中，但佣金引擎读的是 `referrer_id`。

### 环节 3：推荐码验证与绑定

**验证接口**：`POST /v1/referral/verify`（无鉴权，走 referral-visitor 路由）

**后端逻辑**（routes_referral_visitor.js）：
- 查询 `referral_codes` JOIN `users` 获取推荐人信息
- 验证条件：码存在、status=active、未过期、未超使用次数、关联有效 referrer_id
- 返回 `{ valid, code, code_type, referrer_id, referrer_name }`

**绑定接口**：`POST /v1/referral/bind`（需鉴权）

**前端绑定工具**（utils/referral.js `bindByCode()`）：
1. 调用 verifyCode 验证推荐码
2. 检查是否已绑定（hasReferrer）
3. 调用 bindReferrer → `POST /v1/referral/bind` 写入 `user_referrals` 表
4. 更新前端状态（authService、globalData、event 通知）

**改进入口**（pages/mine/mine.js `onJoinMember()`）：
- 扫码后调用 `extractCodeFromScanResult()` 提取推荐码
- 调用 `_doBindAndGoRegister()` 绑定后跳转到建档页
- 绑定失败时传递 `bindResult.reason` 展示具体错误原因（此前修复：2026-06-28）

**状态**：✅ 打通。推荐码验证→绑定→跳转建档页的链路完整，错误提示已优化。

### 环节 4：会员资料填写与建档

**建档接口**：`POST /v1/member/register`（需鉴权）

**前端页面**（pages/register/register.js）：
- 5步表单：基本信息→联系方式→教育职业→择偶偏好→个人介绍
- 支付在第二步（联系方式填写后触发 `onBecomeMember()`）
- 提交时（`onSubmit()`）最终调用 `/v1/member/register` 同步资料

**后端处理**（routes_member.js）：
- 只更新 `users` 表的基本字段（nickname、gender、age、city、phone 等）
- **不修改 role 字段**

**状态**：✅ 打通。建档表单和资料同步正常。

### 环节 5：支付会员建档费

**前端支付入口**：
- register.js 第二步：`onBecomeMember()` → 弹出支付弹窗 → `onConfirmPayment()` → `payment.payRegistration()`
- mine.js：`onJoinMember()` → 扫码绑定 → 跳转 register 页

**支付工具**（utils/payment.js `payRegistration()`）：
1. 调用 `POST /v1/payment/create`（需鉴权），参数 `{ type: 'single_registration', amount: 0.19, description: '会员建档费' }`
2. 调起微信支付 `wx.requestPayment`
3. 支付成功后轮询确认订单状态（`pollPaymentStatus`）
4. 返回 `{ success: true }`

**后端创建订单**（routes_payment.js `POST /create`）：
- 创建 `orders` 记录（status=pending）
- MOCK_PAYMENT 模式：直接标记 paid 并触发佣金+升级
- 真实模式：返回微信支付参数，等待 `/notify` 回调

**金额配置**：限时特惠 0.19 元（原价 199 元，限量 199 个名额）

**状态**：✅ 打通。支付创建和前端调用正常运行。

### 环节 6：支付回调与佣金/角色升级 ⚠️ 关键断点

**支付回调**：`POST /v1/payment/notify`（无鉴权，微信服务端调用）

**处理流程**（routes_payment.js 第400-480行）：
1. 验证微信签名
2. 更新 `orders` 状态为 paid
3. 触发佣金计算 `processCommission()`
4. 调用 `upgradeUserToPaidTier(pool, order.type, order.user_id)` 升级身份

**⚠️ upgradeUserToPaidTier 角色升级函数**（routes_payment.js 第17-80行）：

```javascript
const PAID_TIER_CONFIG = {
  'partner_matchmaker':       { role: 'partner_matchmaker',        codeType: 'creator' },
  'professional_recommender': { role: 'professional_recommender', codeType: 'professional' },
  'city_franchisee':          { role: 'city_franchisee',           codeType: 'city_partner' },
  'community_station':        { role: 'community_station',         codeType: 'community_station' },
};

async function upgradeUserToPaidTier(pool, orderType, userId) {
  const config = PAID_TIER_CONFIG[orderType];
  if (!config) return; // ← single_registration 在此处直接返回！
  // ...
```

**断点分析**：
- `PAID_TIER_CONFIG` 中**不包含 `single_registration`**
- 当用户支付会员建档费后，`upgradeUserToPaidTier` 在第一个 `if (!config)` 即返回
- 用户的 `role` 字段**永远不会**从 `user` 变为 `single` 或其他值
- 前端 register.js 第689行注释写道："之前支付回调已通过 POST /v1/payment/create 更新 role 为 single"，但实际并未执行

**影响范围**：
- 用户 role 始终为 `user`，不产生 `single` 角色
- 前端 register.js 第322行 `getProfile()` 同步用户数据时期待 role 变化，实际上不会有变化
- 推荐官工作台统计 "已注册会员" 使用的是 `orders.type='single_registration' AND status='paid'` 关联查询，不依赖 role 字段，所以统计正确
- 但任何依赖 `role === 'single'` 判断的功能将永远不触发

**状态**：⚠️ 有断点。支付成功，订单状态正确更新为 paid，佣金正常计算，但**用户的 role 字段不会因此改变**。

### 环节 7：前端状态同步

**register.js 支付成功后**（第306-329行）：
1. `authService.setHasProfile(true)` — 标记已建档
2. `authService.setIsPaid(true)` — 标记已付费
3. 调用 `getProfile()` 刷新用户信息
4. `authService.syncUserData(userInfo)` 同步全局状态

**状态**：✅ 打通。前端通过 hasProfile + isPaid 标记实现"会员"状态判断，不依赖 role 字段。

## 四、"会员"身份的判定标准（实际生效机制）

| 维度 | 判定方式 | 生效位置 |
|------|---------|---------|
| 前端"已是会员" | `authService.hasProfile() && authService.isPaid()` | register.js 第316行 `isMemberUnlocked` |
| 前端"访客" | `!hasProfile`（app.js `_syncUserData`） | app.js 第345行 |
| 后端统计"付费会员" | `SELECT COUNT(DISTINCT user_id) FROM orders WHERE type='single_registration' AND status='paid'` | routes_admin.js、routes_stats.js |
| 推荐官"已注册会员" | `user_referrals JOIN users JOIN orders WHERE role='user' AND orders.type='single_registration'` | routes_referral.js 第451行 |

**核心结论**："会员"身份由"是否支付了建档费"界定，而非 role 字段。role 始终为 `user` 是当前系统的设计行为。

## 五、数据一致性风险点

### 风险 1：parent_id 与 referrer_id 双字段问题

登录时通过 scene 参数绑定的推荐人写入 `users.parent_id`，后续通过推荐码绑定的写入 `users.referrer_id`（和 `user_referrals` 表）。佣金计算时 `routes_payment.js` 第267行从 `users.referrer_id` 读取推荐人：

```javascript
const [payers] = await pool.execute('SELECT referrer_id FROM users WHERE id = ?', [req.userId]);
const referrerId = payers[0]?.referrer_id || null;
```

如果用户仅在登录时通过 scene 绑定了推荐人（写入 parent_id），但从未通过推荐码绑定接口设置 referrer_id，那么佣金计算时 `referrer_id` 为 null，推荐人不会获得佣金。

**但** `user_referrals` 表在 `wechat-login` 中也会写入（第158-167行），所以统计层面不会丢失关系数据。佣金引擎的具体实现需要确认是否从 `user_referrals` 表读取。

### 风险 2：user_referrals 唯一约束可能丢失重复绑定

`user_referrals` 表 `user_id` 有 UNIQUE 约束，且 `is_locked = 1` 意味着永久锁定。这确保了每个用户只能绑定一个推荐人，逻辑正确。

### 风险 3：apply_records 申请流程需确认

申请推荐官（如申请公益推荐官）时通过 `POST /v1/apply` 写入 `apply_records` 表。用户需先成为会员（有 paid 订单）才能申请。已修复 2026-06-28 apply.js 中 `updateData` 变量声明顺序问题。

## 七、结论

访客到会员的全业务流程**功能层面已打通**，用户可以完成：扫码进入 → 微信登录 → 验证推荐码 → 绑定推荐关系 → 填写资料 → 支付建档费 → 成为会员 的完整链路。

**唯一的设计差异点**：支付建档费后不修改 role 字段。这是当前架构的主动选择——"会员"身份由订单记录（orders 表）界定，而非 role 字段。所有依赖"是否付费会员"的业务逻辑（统计、推荐官面板）均通过 orders 表查询实现，因此不影响功能。

如果需要统一 role 字段以简化后续逻辑，可在 `upgradeUserToPaidTier` 的 `PAID_TIER_CONFIG` 中增加 `single_registration` 的映射，或单独处理。
