## 首页轮播图点击跳转修复

**问题**：轮播图（尤其是后台配置了 imageUrl 的图片型 slide）点击后无法跳转。

**根因分析**：
1. **主因 — CSS `pointer-events: none` + WXML `catchtap` 死锁**：
   - `.hero-slide-bg` 设置了 `pointer-events: none`（图片不响应触摸）
   - 但 `<image>` 上又有 `catchtap="onHeroSlideTap"`
   - `catchtap` 在 WXML 层捕获了事件（阻止冒泡到父级 `bindtap`）
   - 但 CSS `pointer-events: none` 导致图片的 `catchtap` 处理器不执行
   - 结果：事件被吞掉，父级也收不到 → 完全无响应
2. **次因 — Swiper 手势冲突**：
   - `<swiper>` 的触摸识别可能将 tap 解释为 swipe 起点
   - `bindtap` 容易被 swiper 拦截

**修复内容**（3处改动）：
1. `index.wxml` — 移除 `<image>` 上的 `catchtap`，图片仅作视觉展示
2. `index.wxml` — 父级 `<view>` 的 `bindtap` 改为 `catchtap`，阻止 swiper 拦截
3. `index.js` — `onHeroSlideTap` 增强：300ms 防抖 + 精确 tabBar 匹配 + 错误降级
4. `index.wxss` — 更新 CSS 注释说明新架构

## 首页轮播图点击跳转修复（第二轮）

**问题**：第一轮修复后仍然无效，点击轮播图依旧无法跳转。

**深层根因**：
1. **主因 — WeChat `<swiper>` 手势拦截子元素 tap 事件**：
   - 微信小程序的 `<swiper>` 组件会在触摸层拦截子元素（包括 `<view>`）上的 tap 事件
   - 即使使用 `catchtap`，swiper 的触摸手势识别器也会优先消费 tap → 内部 view 收不到事件
   - **解决**：将 `bindtap` 直接绑定在 `<swiper-item>` 上（原生组件级别），绕过 swiper 的子元素事件拦截
2. **次因 — `wx.switchTab`/`wx.navigateTo` 不返回 Promise**：
   - 代码中 `nav.catch(...)` 调用在 undefined 上，触发 TypeError
   - 虽然导航 API 已被调用，但未捕获的错误在特定场景下可能导致异常行为
   - **解决**：改用 `fail` 回调参数，这是 wx API 规范的错误处理方式

**第二轮修复内容**（3处改动）：
1. `index.wxml` — `bindtap` 从子 `<view>` 移到 `<swiper-item>` 上（关键修复）
2. `index.js` — `wx.switchTab`/`wx.navigateTo` 改用 `fail` 回调，添加 console.log 调试日志
3. `index.wxss` — 更新注释

## 首页轮播图点击失效根因分析（第三轮—真因）

**关键线索**：管理后台配置轮播图之前可以点击跳转，配置之后不行。

**真因**：管理后台保存轮播图时，**所有 slide 的 `ctaPage` 字段均为空字符串**。

数据库 `config` 表中 `hero_slides` 存储的 JSON：
```json
[
  {"id":"love","title":"人人推荐，人人美好","ctaPage":"",...},
  {"id":"salon","title":"六人轻快沙龙","ctaPage":"",...},
  {"id":"charity","title":"身边皆是红娘","ctaPage":"",...},
  {"id":"community","title":"人人媒好AI脱单","ctaPage":"",...}
]
```

**对比默认数据**（配置前使用）：
| id | ctaPage |
|----|---------|
| salon | `/subpackages/activity/pages/salon-list/salon-list` |
| ai-match | `/pages/avatar/avatar` |
| matchmaker | `/pages/matchmaker/matchmaker` |

**数据流链路**：
1. 管理后台 `PUT /admin-api/config` → `hero_slides` = JSON.stringify(slides)
2. Prisma `upsert` 存入 MySQL `config` 表
3. 小程序 `GET /v1/config/map` → 读取 `hero_slides` → `JSON.parse()`
4. 小程序发现 ctaPage 为空 → `!page` → return 不跳转

**修复（第三轮）**（3处改动）：
1. `index.js` `_initHeroSlides` — 新增 ctaPage 三层兜底逻辑：
   - 按 slide.id 精确匹配默认跳转
   - 按标题关键词模糊匹配（沙龙→活动页、AI→画像页、推荐官/红娘→推荐官页）
   - 无匹配兜底 → 首页
2. `admin settings/index.tsx` — 保存时检测空 ctaPage 并 toast 警告
3. 建议用户在管理后台补填每张轮播图的"跳转页面"字段

## 推荐官自荐功能分析（推荐自己成为单身会员）

### 系统存在 4 层独立的防自荐机制：

### 支付回调的特殊处理（payment.ts ~428）

当推荐官为自己购买会员建档（`type: "single_registration"`）时：
- ✅ 系统会**记录**一条 `ReferralUsageLog`（scene: "self_register"）用于数据分析
- ❌ 系统**不会**生成佣金记录
- ❌ 系统**不会**修改 `recommendedBy` 字段
- 这纯粹是**埋点日志**，不是业务收益

### 实际业务流程

```
推荐官尝试自荐:
  推荐官用自己的推荐码 → referral.ts 拦截 → ❌ "不能自己推荐自己"

推荐官登记为会员（无自荐，由他人推荐）:
  推荐官被A推荐 → A得99元佣金 → 推荐官自己成为付费会员 → 推荐官不得佣金
  推荐官已有推荐人B → 建档后B得99元 → 推荐官不得佣金

推荐官推荐他人:
  推荐官分享推广码 → 新用户C扫码建档付费 → ✅ 推荐官得99元
```

### 所有推荐官身份均适用此规则

`public_matchmaker` / `partner_matchmaker` / `professional_recommender` / `community_station` / `city_franchisee` 均不能自荐，防自荐检测是角色无关的（基于 userId 比较）。
