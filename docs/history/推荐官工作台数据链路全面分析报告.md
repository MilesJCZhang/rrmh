# 推荐官工作台数据链路全面分析报告

**分析日期**: 2026-06-20
**分析范围**: 小程序推荐官工作台（公益推荐官/联创推荐官/专业推荐官）所有业务数据统计展示、推荐人员信息统计、数据链路打通情况

## 一、推荐官工作台展示的业务数据统计总览

### 1.1 主工作台页面（subpackages/matchmaker/pages/matchmaker-workbench/）

三套文件共同构成工作台入口：
- `matchmaker-workbench.js` — 主包版本
- `subpackages/matchmaker/pages/matchmaker-workbench/matchmaker-workbench.js` — 分包增强版（当前生产使用）
- `subpackages/matchmaker/pages/recommender/recommender.js` — 专业推荐官专用

展示数据分为 4 个区块：

| 区块 | 数据字段 | 数据来源 API | 当前状态 |
|------|---------|-------------|:-------:|
| 收益概览 | `todayIncome`, `monthIncome`, `totalIncome`, `withdrawable` | `GET /v1/income/summary` | ⚠️ 字段不匹配 |
| 业务统计 | `memberStats.count/income`, `partnerStats.count/income`, `salonStats.count/income` | `GET /v1/income/stats` | ⚠️ 缺少金额 |
| 数据看板 | `partner_matchmaker_count`, `public_matchmaker_count`, `registered_member_count`, `visitor_count` | `GET /v1/referral/workbench-stats` | ✅ 完全打通 |
| 分类明细 | 各类型下的用户名册分页列表 | `GET /v1/referral/workbench-detail` | ⚠️ 缺 status 字段 |
| 收益记录 | 最近收益列表（icon/title/time/amount/settled） | `GET /v1/income/records` | ⚠️ 字段名不匹配 |
| 洞察数据 | `insightStats`, `insightReferredUsers`, `insightReferralChain` | `GET /v1/referral/my-insight` | ❌ 后端仅返回计数 |

### 1.2 "我的会员"页面（subpackages/matchmaker/pages/matchmaker/earnings.js）

展示推荐官旗下会员列表，数据来源为 `GET /v1/referral/my-insight`，读取其 `referral_chain` 字段。但该接口仅返回 `{ total_referrals, month_referrals }`，不包含链条数据。

## 二、逐接口数据链路分析

### 接口 1：GET /v1/referral/my-insight（推荐洞察）

**后端实现** (routes_referral.js L332-361):
```javascript
// 只返回两个字段
data: {
  total_referrals: 总推荐人数,
  month_referrals: 本月新增推荐人数
}
```

**前端期望** (matchmaker-workbench.js L138-154):
```javascript
insight.code_info    // 推荐码信息 → 设为 insightCodeInfo
insight.stats        // 统计数据 → 设为 insightStats
insight.referred_users  // 被推荐用户列表 → 设为 insightReferredUsers（取前10）
insight.referral_chain  // 推荐链条 → 设为 insightReferralChain
```

**前端期望** (recommender.js L189-196):
```javascript
insight.stats.total_referred   // 总推广数
insight.code_info.code         // 推广码
insight.code_info.created_at   // 推广码创建时间
insight.referred_users.length  // 推荐用户数
```

**结论：❌ 严重不匹配**
后端只返回计数，前端期望完整对象。由于前端使用 `|| {}` 兜底，不会崩溃，但：
- 推荐官工作台 WXML 不渲染 insight 数据（仅设 data，未在模板使用），无可见影响
- 专业推荐官页面 (`recommender.js`) 的 `totalPromotions` 无法获取真实值，回退为 `referredUsers.length`（空数组，显示 0）
- 推荐码信息 (`code_info`) 为空，推广链接生成可能异常

### 接口 2：GET /v1/income/summary（收益概览）

**后端实现** (routes_income.js L17-64):
```javascript
data: {
  withdrawable: 可提现余额,
  total: 累计总收入,       // 注意：字段名是 total
  withdrawn: 已提现金额,
  frozen: 冻结中金额
}
// 不包含 todayIncome、monthIncome
```

**前端期望** (matchmaker-workbench.js L143-146):
```javascript
summary?.todayIncome     // 今日收益 → 后端无 → 始终显示 0.00
summary?.monthIncome     // 本月收益 → 后端无 → 始终显示 0.00
summary?.totalIncome     // 累计收益 → 后端字段名是 total → 始终显示 0.00
summary?.withdrawable    // 可提现   → ✅ 匹配
```

**结论：⚠️ 字段缺失与命名不匹配**
- 今日收益、本月收益：后端未计算时间维度的收益拆分，前端始终显示 ¥0.00
- 累计收益：后端返回 `total`，前端读取 `totalIncome`，永远取不到值
- 可提现余额：唯一正确打通的字段

### 接口 3：GET /v1/income/stats（业务统计）

**后端实现** (routes_income.js L119-176):
```javascript
data: {
  memberCount: 建档数,      // 通过推荐关系绑定 + 已支付 single_registration 的用户
  partnerCount: 联创推荐官数,  // 通过推荐关系 + 角色为 partner_matchmaker
  salonCount: 沙龙参与次数   // salon_group_members checked_in 或 activity_registrations
}
// 不包含任何收入金额
```

**前端期望** (matchmaker-workbench.js L147-149):
```javascript
stats?.memberCount    → ✅ 匹配
stats?.memberIncome   → 后端无此字段 → 始终显示 0.00
stats?.partnerCount   → ✅ 匹配
stats?.partnerIncome  → 后端无此字段 → 始终显示 0.00
stats?.salonCount     → ✅ 匹配
stats?.salonIncome    → 后端无此字段 → 始终显示 0.00
```

**结论：⚠️ 统计数据打通但收益金额缺失**
- 三项业务的数量统计均正确从 user_referrals / users / orders / salon_group_members 表查询
- 但对应的收入金额（memberIncome/partnerIncome/salonIncome）后端未计算，前端始终显示 0.00

### 接口 4：GET /v1/income/records（收益记录列表）

**后端实现** (routes_income.js L67-116):
```javascript
list: [{
  id, type, amount, rate, status, orderId, remark, createdAt, confirmedAt
}]
```

**WXML 期望** (matchmaker-workbench.wxml L298-319):
```xml
item.icon      // 图标 → 后端无 → 不显示
item.title     // 标题 → 后端无 → 不显示  
item.time      // 时间 → 后端字段名是 createdAt → 不显示
item.amount    // 金额 → ✅ 匹配
item.settled   // 是否已结算 → 后端字段名是 status → 不显示
```

**结论：⚠️ 字段名完全不一致**
后端返回原始数据库字段名（`type`/`createdAt`/`status`），WXML 模板使用显示友好字段名（`icon`/`title`/`time`/`settled`）。Mock 模式下由 `income.service.js` 的 `_MOCK` 数据提供正确字段，切换生产环境后收益记录列表将**一片空白**（只有金额能显示）。

### 接口 5：GET /v1/referral/workbench-stats（数据看板统计）

**后端实现** (routes_referral.js L436-493):
```sql
-- 1. partner_matchmaker: user_referrals JOIN users WHERE role='partner_matchmaker'
-- 2. public_matchmaker: user_referrals JOIN users WHERE role='public_matchmaker'  
-- 3. registered_member: user_referrals JOIN users JOIN orders WHERE role='user' AND type='single_registration' AND status='paid'
-- 4. visitor: visitor_logs WHERE reg_status='pending'
```

**前端期望** (matchmaker-workbench.js L155-160):
```javascript
workbenchStats.partner_matchmaker_count   → ✅
workbenchStats.public_matchmaker_count    → ✅
workbenchStats.registered_member_count    → ✅
workbenchStats.visitor_count              → ✅
```

**结论：✅ 完全打通**
四项统计数据的前后端字段名完全一致，数据查询逻辑正确（联创推荐官和公益推荐官区分 role 字段，已注册会员通过 orders 表验证支付状态，访客通过 visitor_logs 表统计）。

### 接口 6：GET /v1/referral/workbench-detail（分类明细列表）

**后端实现** (routes_referral.js L501-657):

4 种类型分别查询并格式化输出，字段名与前端匹配：

| 类型 | 查询逻辑 | 返回字段 | 前端渲染字段 |
|------|---------|---------|------------|
| partner_matchmaker | user_referrals JOIN users WHERE role='partner_matchmaker' | id, name, nickname, avatar, role, joinTime, createdAt | id, name, joinTime, **status** ⚠️ |
| public_matchmaker | 同上 WHERE role='public_matchmaker' | 同上 | id, name, createdAt |
| registered_member | 同上 JOIN orders WHERE type='single_registration' AND status='paid' | id, name, nickname, avatar, role, joinTime, createdAt | id, name, createdAt |
| visitor | visitor_logs WHERE reg_status='pending' | id, visitor_nickname, visitor_avatar, visit_time | id, visitor_nickname, visit_time |

**结论：⚠️ partner_matchmaker 缺少 status 字段**
- WXML L159-160 根据 `item.status === 'active'` 显示"已激活"或"未激活"
- 后端格式化输出中未包含 status，实际应该从 users 表的 status 字段读取
- 后端 SQL 查询已 JOIN users 表但 SELECT 未包含 u.status，格式化函数也遗漏此字段
- 其他三种类型的明细字段基本匹配

### 接口 7：GET /v1/member/list（会员列表）

**后端实现** (routes_member.js L70-111):
```javascript
data: {
  list: [{ id, nickname, avatar, gender, age, role, bind_time, created_at }],
  total, page, page_size, has_more
}
```
从 user_referrals JOIN users 查询，按推荐关系绑定。

**结论：✅ 接口实现完整**
但 earnings.js 的"我的会员"标签页并未调用此接口，而是错误地调用了 `/v1/referral/my-insight`（见下文问题 3）。

## 三、数据链路断点汇总

### 问题 1：收益金额全部缺失（影响面：P0）

影响页面：推荐官工作台收益概览区（今日/本月/累计）+ 业务统计区（建档收入/联创收入/沙龙收入）

根因：`routes_income.js` 的 `/summary` 和 `/stats` 接口均只返回计数和可提现金额，未按时间维度（今日/本月）拆分收益，也未计算各业务类型的累计收入金额。

当前表现：开发模式 Mock 数据显示正常，切换生产环境后全部显示 ¥0.00 或 0。

### 问题 2：收益记录字段映射断裂（影响面：P0）

影响页面：推荐官工作台"最近收益记录"区块

根因：后端 `commission_records` 表字段名（`type`/`createdAt`/`status`）与 WXML 模板期望的显示字段名（`icon`/`title`/`time`/`settled`）不一致，缺少数据转换层。

当前表现：Mock 数据显示正常，生产环境记录列表只显示金额数值，不显示图标、标题、时间、结算状态。

### 问题 3："我的会员"数据源错误（影响面：P0）

影响页面：earnings.js 的"我的会员"标签页

根因：`loadMembers()` 调用 `/v1/referral/my-insight` 并读取 `res.referral_chain`，但该接口只返回 `{ total_referrals, month_referrals }` 两个数字。正确的数据源应该是 `/v1/member/list`（已实现且完整）。

当前表现：会员列表始终为空，totalMembers 显示 0。

### 问题 4：推荐洞察数据贫血（影响面：P1）

影响页面：专业推荐官页面（recommender.js）

根因：`/v1/referral/my-insight` 只返回两个计数，前端期望完整的推荐码信息、统计数据、推荐用户列表、推荐链条。

当前表现：`totalPromotions` 始终为 0，推广码信息无法显示。

### 问题 5：联创推荐官明细缺激活状态（影响面：P1）

影响页面：工作台数据看板 → 联创推荐官明细展开列表

根因：`routes_referral.js` workbench-detail 的 partner_matchmaker 格式化输出未包含 `status` 字段。

当前表现：展开列表的状态列始终显示"未激活"（因为 `item.status` 为 undefined，走 `status-inactive` 样式）。

### 问题 6：今日/本月收益无法区分（影响面：P2）

影响页面：收益概览首行（今日收益/本月收益）

根因：`/v1/income/summary` 使用 `commission_records` 表计算总收入，但未按时间维度过滤。需增加 `created_at >= today` 和 `created_at >= month_start` 的条件查询。

## 六、附录：Mock 数据影响说明

当前 `services/income.service.js` 中 `DEV_MOCK_DATA` 控制是否使用 Mock 数据：

- Mock 模式下：收益概览、业务统计、最近记录均有硬编码数据，字段名与 WXML 完全匹配，所有展示看起来正常
- 生产模式下：上述 P0 问题全部暴露，收益相关数据显示异常

配置文件位置：`utils/config.js` 中的 `DEV_MOCK_DATA` 常量。

建议：在修复上述 P0/P1 问题前，生产环境暂不切换 `DEV_MOCK_DATA = false`。
