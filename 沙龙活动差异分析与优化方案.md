# 相亲小程序沙龙活动差异分析与优化方案

## 文档信息
- **创建日期**: 2026-06-06
- **版本**: v1.0
- **目标**: 全面分析常规沙龙与推荐官沙龙的差异，提出系统性优化方案

---

## 一、当前实现分析

### 1.1 数据结构对比

#### 常规沙龙（3男3女分组模式）
- **核心字段**:
  - `type`: 'mixed'（默认值）
  - `is_grouped`: 1（启用分组）
  - `max_participants`: 6（默认）
  - `max_per_gender`: 3（每性别上限）
  - `score_tier`: 'silver'（评分等级要求）
  - `allowed_tiers`: 'gold,silver,bronze'
  
- **分组机制**:
  - 使用 `salon_groups` 表进行评分段分组
  - 每组3男3女，按评分段（gold/silver/bronze）匹配
  - 使用 `salon_group_members` 表记录成员

#### 推荐官沙龙（男场/女场）
- **核心字段**:
  - `type`: 'male_salon' 或 'female_salon'
  - `is_grouped`: 0（不分组）
  - `max_recommenders`: 9（推荐官上限）
  - `max_companions_per_person`: 2（每人最多随行人数）
  - `total_cap`: 27（单场总人数封顶）
  - `score_tier`: 'all'（不限制评分等级）
  - `week_day`: 1-7（每周几）
  
- **特殊限制**:
  - 强限制性别：男推荐官只能报男场，女推荐官只能报女场
  - 每周报名次数限制：每人每周仅限报名1次
  - 随行人员需详细资料（姓名、手机、性别、年龄、行业等）

### 1.2 功能差异对比表

| 对比维度 | 常规沙龙 | 推荐官沙龙 |
|---------|----------|------------|
| **目标用户** | 所有用户（按评分等级限制） | 推荐官（按性别限制） |
| **分组模式** | 3男3女精准分组 | 单性别，不分组 |
| **人数限制** | 固定6人（3男3女） | 27人封顶（9推荐官×3人） |
| **报名权限** | 需要评分等级，按tier限制 | 不限制评分，限制性别 |
| **每周限制** | 无 | 每人每周仅限报名1次 |
| **随行人员** | 可选，最多3人 | 可选，最多2人，需详细资料 |
| **评分等级筛选** | 有（全部/优质80+/良好60+/基础<60） | 无 |
| **UI主题色** | 红色（#C8102E） | 男场蓝色（#1565C0）/女场粉色（#C2185B） |
| **服务保障** | 3男3女精准分组、资料隐私加密保护 | 可携随行朋友、按评分段匹配 |
| **审核流程** | 需要审核 | 需要审核，审核通过后生成海报 |
| **收益模式** | 报名费 | 报名费 + 推荐官收益 |

### 1.3 报名流程差异

#### 常规沙龙报名流程
```
用户点击"立即报名" 
  ↓
检查用户是否登录
  ↓
检查沙龙状态（published/open）
  ↓
检查用户评分等级是否符合要求
  ↓
弹出报名资料填写弹窗
  ↓
提交报名（更新salons表和salon_group_members表）
```

#### 推荐官沙龙报名流程
```
用户点击"立即报名" 
  ↓
检查用户是否登录
  ↓
检查沙龙状态（published/open）
  ↓
检查用户性别是否匹配沙龙类型（男场/女场）
  ↓
检查每周报名次数限制（通过weekly_signup表）
  ↓
检查人数限额（27人封顶）
  ↓
弹出报名资料填写弹窗（含详细随行人员资料）
  ↓
提交报名
  ↓
记录每周报名信息到weekly_signup表
  ↓
如果满27人，自动更新沙龙状态为'full'
```

### 1.4 当前实现存在的问题

#### 问题1：代码逻辑复杂，维护困难
- **现象**: `salon-detail.js` 中有大量 `if (type)` 判断，代码可读性差
- **影响**: 新增功能时需要同时修改多个地方，容易引入bug
- **示例**:
  ```javascript
  // 当前代码中有大量这样的判断
  const salonRemain = type ? recommenderRemain : (maleRemain + femaleRemain);
  if (type) {
    // 推荐官沙龙逻辑
  } else {
    // 常规沙龙逻辑
  }
  ```

#### 问题2：数据模型耦合严重
- **现象**: 常规沙龙和推荐官沙龙共用 `salons` 表，但通过 `type` 字段区分
- **影响**: 
  - 数据库字段冗余（常规沙龙不需要 `max_recommenders`，推荐官沙龙不需要 `is_grouped`）
  - 查询时需要频繁判断 `type`，影响性能
  - 扩展性差，新增沙龙类型需要修改大量代码

#### 问题3：UI展示不一致
- **现象**: 两种沙龙的详情页结构相似但细节差异大，导致代码重复
- **影响**: 
  - 样式维护困难
  - 用户体验不一致
  - 适配新需求时需要双倍工作量

#### 问题4：报名流程差异处理不当
- **现象**: 推荐官沙龙的27人封顶计算逻辑复杂，容易出错
- **影响**: 
  - 可能出现超员问题
  - 每周报名次数限制的实现依赖前端判断，安全性不足
  - 取消报名时释放名额的逻辑可能不完整

#### 问题5：权限控制不够精细
- **现象**: 推荐官沙龙的性别限制仅在前端判断，后端校验可能不完整
- **影响**: 
  - 可能存在越权报名的安全问题
  - 数据一致性无法保证

#### 问题6：用户体验问题
- **现象**: 
  - 推荐官沙龙报名时需要填写详细资料，流程较长
  - 常规沙龙的评分等级限制可能导致部分用户无法参与
  - 两种沙龙的列表页筛选逻辑不同，用户可能困惑
- **影响**: 转化率降低，用户流失

## 二、优化方案

### 2.1 架构优化：解耦数据模型

#### 方案：使用继承+扩展表设计

**目标**: 将常规沙龙和推荐官沙龙的数据模型解耦，提高可维护性和扩展性。

**实施步骤**:

1. **保留 `salons` 表作为基础表**，包含所有沙龙类型的公共字段：
   ```sql
   CREATE TABLE salons (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     title TEXT NOT NULL,
     description TEXT,
     cover_image TEXT,
     type TEXT NOT NULL DEFAULT 'mixed', -- 'mixed' | 'male_salon' | 'female_salon'
     location TEXT,
     city TEXT,
     event_date TEXT NOT NULL,
     start_time TEXT,
     end_time TEXT,
     registration_fee INTEGER DEFAULT 399,
     status TEXT DEFAULT 'draft',
     organizer_id INTEGER,
     created_at TEXT DEFAULT CURRENT_TIMESTAMP,
     updated_at TEXT DEFAULT CURRENT_TIMESTAMP
   );
   ```

2. **创建扩展表**，分别存储两种沙龙的特有字段：

   **常规沙龙扩展表**:
   ```sql
   CREATE TABLE salon_mixed_config (
     salon_id INTEGER PRIMARY KEY,
     is_grouped INTEGER DEFAULT 1,
     max_participants INTEGER DEFAULT 6,
     max_per_gender INTEGER DEFAULT 3,
     score_tier TEXT DEFAULT 'all',
     allowed_tiers TEXT DEFAULT 'gold,silver,bronze',
     FOREIGN KEY (salon_id) REFERENCES salons(id)
   );
   ```

   **推荐官沙龙扩展表**:
   ```sql
   CREATE TABLE salon_gender_config (
     salon_id INTEGER PRIMARY KEY,
     max_recommenders INTEGER DEFAULT 9,
     max_companions_per_person INTEGER DEFAULT 2,
     total_cap INTEGER DEFAULT 27,
     week_day INTEGER,
     FOREIGN KEY (salon_id) REFERENCES salons(id)
   );
   ```

3. **创建视图**，简化查询：
   ```sql
   CREATE VIEW v_salon_detail AS
   SELECT s.*, 
          smc.*,  -- 常规沙龙配置
          sgc.*   -- 推荐官沙龙配置
   FROM salons s
   LEFT JOIN salon_mixed_config smc ON s.id = smc.salon_id AND s.type = 'mixed'
   LEFT JOIN salon_gender_config sgc ON s.id = sgc.salon_id AND s.type IN ('male_salon', 'female_salon');
   ```

**优势**:
- 数据模型清晰，职责分离
- 扩展性好，新增沙龙类型只需添加扩展表
- 查询性能优化，减少不必要的字段读取

### 2.2 代码优化：组件化与策略模式

#### 方案：使用策略模式处理差异逻辑

**目标**: 将两种沙龙的差异逻辑封装到不同的策略类中，提高代码可维护性。

**实施步骤**:

1. **创建沙龙策略基类**:
   ```javascript

   class SalonStrategy {
     constructor(salon) {
       this.salon = salon;
     }
     
     // 获取剩余名额
     getRemainCount() {
       throw new Error('子类必须实现此方法');
     }
     
     // 检查报名权限
     checkSignupPermission(user) {
       throw new Error('子类必须实现此方法');
     }
     
     // 获取报名表单配置
     getSignupFormConfig() {
       throw new Error('子类必须实现此方法');
     }
     
     // 提交报名
     async submitSignup(formData) {
       throw new Error('子类必须实现此方法');
     }
   }
   
   module.exports = SalonStrategy;
   ```

2. **实现常规沙龙策略**:
   ```javascript
   // services/salon-strategies/mixed.strategy.js
   const SalonStrategy = require('./base.strategy');
   
   class MixedSalonStrategy extends SalonStrategy {
     getRemainCount() {
       return this.salon.maleRemain + this.salon.femaleRemain;
     }
     
     checkSignupPermission(user) {
       if (this.salon.score_tier !== 'all') {
         const allowedTiers = this.salon.allowed_tiers.split(',');
         if (!allowedTiers.includes(user.scoreTier)) {
           return { canSignup: false, reason: '评分等级不符合要求' };
         }
       }
       return { canSignup: true };
     }
     
     getSignupFormConfig() {
       return {
         showGenderSelect: true,
         maxCompanions: 3,
         requireScoreTier: true
       };
     }
   }
   
   module.exports = MixedSalonStrategy;
   ```

3. **实现推荐官沙龙策略**:
   ```javascript
   // services/salon-strategies/gender.strategy.js
   const SalonStrategy = require('./base.strategy');
   
   class GenderSalonStrategy extends SalonStrategy {
     getRemainCount() {
       const maxRecommenders = this.salon.max_recommenders || 9;
       const currentCount = this.salon.type === 'male_salon' 
         ? this.salon.male_count 
         : this.salon.female_count;
       return maxRecommenders - currentCount;
     }
     
     checkSignupPermission(user) {
       // 检查性别

         return { canSignup: false, reason: '此为男推荐官专属沙龙' };
       }
       if (this.salon.type === 'female_salon' && user.gender !== 'female') {
         return { canSignup: false, reason: '此为女推荐官专属沙龙' };
       }
       
       // 检查每周报名次数
       // ...（后端校验）
       
       return { canSignup: true };
     }
     
     getSignupFormConfig() {
       return {
         showGenderSelect: false, // 自动填充，不可修改
         maxCompanions: 2,
         requireDetailedInfo: true // 需要详细资料
       };
     }
   }
   
   module.exports = GenderSalonStrategy;
   ```

4. **在详情页中使用策略**:
   ```javascript
   // salon-detail.js
   const MixedSalonStrategy = require('../../services/salon-strategies/mixed.strategy');
   const GenderSalonStrategy = require('../../services/salon-strategies/gender.strategy');
   
   function getStrategy(salon) {
     if (salon.type === 'mixed') {
       return new MixedSalonStrategy(salon);
     } else {
       return new GenderSalonStrategy(salon);
     }
   }
   
   // 在 loadDetail 中使用
   const strategy = getStrategy(salon);
   const remainCount = strategy.getRemainCount();
   const permission = strategy.checkSignupPermission(userInfo);
   ```

**优势**:
- 代码逻辑清晰，易于维护
- 新增沙龙类型只需添加新的策略类
- 便于单元测试

### 2.4 报名流程优化

#### 优化1：加强后端权限校验

**问题**: 当前推荐官沙龙的性别限制主要在前端判断，存在安全风险。

**解决方案**:
1. 在后端API中添加严格的权限校验：
   ```javascript
   // routes_salon.js - 报名接口
   function checkSignupPermission(salon, userId) {
     // 1. 检查沙龙状态
     if (!['published', 'open'].includes(salon.status)) {
       return { error: '该沙龙暂未开放报名' };
     }
     
     // 2. 检查用户是否已报名
     const existing = db.prepare(
       'SELECT id FROM salon_group_members WHERE salon_id = ? AND user_id = ? AND status != "cancelled"'
     ).get(salon.id, userId);
     if (existing) {
       return { error: '您已报名该沙龙' };
     }
     
     // 3. 推荐官沙龙：检查性别
     if (salon.type === 'male_salon' || salon.type === 'female_salon') {
       const user = db.prepare('SELECT gender FROM users WHERE id = ?').get(userId);
       if (salon.type === 'male_salon' && user.gender !== 'male') {
         return { error: '此为男推荐官专属沙龙，仅男士可报名' };
       }
       if (salon.type === 'female_salon' && user.gender !== 'female') {
         return { error: '此为女推荐官专属沙龙，仅女士可报名' };
       }
       
       // 4. 检查每周报名次数
       const weekMonday = getMonday(salon.event_date);
       const weeklySignup = db.prepare(
         'SELECT id FROM weekly_signup WHERE user_id = ? AND salon_week_day = ?'
       ).get(userId, weekMonday);
       if (weeklySignup) {
         return { error: '本周报名次数已用尽（每周仅限报名1次）' };
       }
     }
     
     return { success: true };
   }
   ```

#### 优化2：优化27人封顶计算逻辑

**问题**: 当前27人封顶的计算逻辑可能不准确，需要考虑随行人员的动态变化。

**解决方案**:
1. 创建数据库触发器或事务，确保人数计算的原子性：
   ```javascript
   // 使用事务确保数据一致性
   function signupGenderSalon(salonId, userId, companions) {
     const result = db.transaction(() => {
       // 1. 锁定沙龙记录，防止并发
       const salon = db.prepare('SELECT * FROM salons WHERE id = ?').get(salonId);
       
       // 2. 计算当前总人数
       const countResult = db.prepare(`
         SELECT 
           COUNT(*) as main_count,
           SUM(CASE WHEN companions_json IS NOT NULL THEN json_array_length(companions_json) ELSE 0 END) as companion_count
         FROM salon_group_members 
         WHERE salon_id = ? AND status != 'cancelled'
       `).get(salonId);
       
       const currentTotal = countResult.main_count + (countResult.companion_count || 0);
       const newTotal = currentTotal + 1 + (companions ? companions.length : 0);
       
       // 3. 检查是否超限
       if (newTotal > salon.total_cap) {
         throw new Error(`本场沙龙已满员（${salon.total_cap}人封顶）`);
       }
       
       // 4. 插入报名记录
       db.prepare(`
         INSERT INTO salon_group_members (salon_id, user_id, gender, companions_json, status)
         VALUES (?, ?, (SELECT gender FROM users WHERE id = ?), ?, 'registered')
       `).run(salonId, userId, userId, JSON.stringify(companions));
       
       // 5. 更新沙龙人数
       const genderField = salon.type === 'male_salon' ? 'male_count' : 'female_count';
       db.prepare(`UPDATE salons SET ${genderField} = ${genderField} + 1 WHERE id = ?`).run(salonId);
       
       // 6. 如果满员，更新状态
       if (newTotal + 1 >= salon.total_cap) {
         db.prepare("UPDATE salons SET status = 'full' WHERE id = ?").run(salonId);
       }
       
       // 7. 记录每周报名
       const weekMonday = getMonday(salon.event_date);
       db.prepare(`
         INSERT INTO weekly_signup (user_id, salon_id, salon_week_day)
         VALUES (?, ?, ?)
       `).run(userId, salonId, weekMonday);
       
       return { success: true };
     })();
     
     return result;
   }
   ```

#### 优化3：简化报名资料填写流程

**问题**: 推荐官沙龙报名时需要填写详细资料，流程较长，可能导致用户放弃。

**解决方案**:
1. 分步填写，第一步仅填必填项（姓名、手机），第二步可选填详细资料
2. 自动填充用户已有资料
3. 提供"快速报名"选项，仅填核心信息

   ```javascript
   // 分步表单配置
   const signupSteps = {
     mixed: [
       { title: '基本信息', fields: ['name', 'mobile', 'gender'] },
       { title: '随行人员', fields: ['companions'] }
     ],
     gender: [
       { title: '基本信息', fields: ['name', 'mobile'] },
       { title: '详细资料', fields: ['age', 'industry', 'identity'], optional: true },
       { title: '随行人员', fields: ['companions'] }
     ]
   };
   ```

### 2.5 用户体验优化

#### 优化1：统一列表页筛选逻辑

**问题**: 常规沙龙有评分等级筛选Tab，推荐官沙龙没有，用户可能困惑。

**解决方案**:
1. 为推荐官沙龙添加"推荐官等级"筛选（如：联创推荐官、专业推荐官等）
2. 或者，统一移除筛选Tab，改为更直观的标签展示

   ```xml
   <!-- 推荐官沙龙列表项添加标签 -->
   ```

   ```javascript
   // 后端定时任务：发送提醒
   // 每天定时检查即将开始的沙龙，发送提醒
   function sendSalonReminder() {
     const upcomingSalons = db.prepare(`
       SELECT s.*, GROUP_CONCAT(sm.user_id) as user_ids
       FROM salons s
       JOIN salon_group_members sm ON s.id = sm.salon_id
       WHERE s.event_date = date('now', '+1 day')
       AND s.status = 'published'
       AND sm.status = 'registered'
     `).all();
     
     for (const salon of upcomingSalons) {
       const userIds = salon.user_ids.split(',');
       for (const userId of userIds) {
         // 发送微信订阅消息
         sendWechatMessage(userId, 'salon_reminder', {
           salonName: salon.title,
           salonTime: `${salon.event_date} ${salon.start_time}`
         });
       }
     }
   }
   ```

2. 在前端axios拦截器中统一处理：
   ```javascript
   // utils/request.js
   function handleResponse(response) {
     const res = response.data;
     if (res.code === 0 || res.code === 200) {
       return res.data;
     } else {
       throw new Error(res.message || '操作失败');
     }
   }
   ```

#### 优化2：优化数据库查询性能

**问题**: 沙龙列表查询可能涉及多表关联，性能较差。

**解决方案**:
1. 使用索引优化查询：
   ```sql
   CREATE INDEX idx_salons_type_status ON salons(type, status);
   CREATE INDEX idx_salons_event_date ON salons(event_date);
   CREATE INDEX idx_salon_group_members_salon_user ON salon_group_members(salon_id, user_id);
   ```

2. 使用缓存减少数据库查询：
   ```javascript
   // 缓存沙龙列表（5分钟有效期）
   const salonListCache = new Map();
   
   function getSalonListWithCache(params) {
     const cacheKey = JSON.stringify(params);
     const cached = salonListCache.get(cacheKey);
     
     if (cached && Date.now() - cached.timestamp < 5 * 60 * 1000) {
       return cached.data;
     }
     
     const data = getSalonListFromDB(params);
     salonListCache.set(cacheKey, {
       data,
       timestamp: Date.now()
     });
     
     return data;
   }
   ```
